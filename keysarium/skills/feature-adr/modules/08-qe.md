# Step 8: Quality Engineering

> Verify the implementation through testing, review, and validation.

## When

Always runs. Adapts depth by tier:
- **S:** Smoke test — compile + existing tests pass
- **M:** Unit tests + basic code review
- **L:** Unit + integration tests + thorough review
- **XL:** Full QE — unit + integration + e2e + security + performance review

## Model

sonnet (analytical evaluation)

## Input

- `{CODE_CHANGES}` from Step 7
- `{REQUIREMENTS}` from Step 1 (for acceptance criteria)
- `{ARCHITECTURE}` from Step 5 (for structural validation)
- `{IMPL_PLAN}` from Step 6 (for completeness check)

## Protocol

### 1. Smoke Tests (All tiers)

```
- [ ] Code compiles / no syntax errors
- [ ] Existing test suite passes (no regressions)
- [ ] Linter passes (if configured)
- [ ] Type checker passes (if configured)
```

For S-tier: if all pass, proceed to checkpoint. Done.

### 2. Test Generation (M+)

#### Unit Tests
For each new/modified module:
- Test happy path
- Test edge cases (null, empty, boundary values)
- Test error paths
- Follow existing test patterns in codebase

#### Integration Tests (L/XL)
For each integration point:
- Test API endpoints with real request/response cycle
- Test database operations with test database
- Test message queue interactions
- Test external service mocks

#### E2E Tests (XL only)
For critical user flows:
- Test complete user journey end-to-end
- Test with realistic data volumes
- Test failure recovery scenarios

### 3. Code Review via Brutal Honesty Review (M+)

> Load: `references/agentic-qe/brutal-honesty-review.md`

Apply the **brutal-honesty-review** protocol from agentic-qe (Tier 2: Validated).

#### Mode Selection

Choose review mode based on context:

| Mode | When | Style |
|------|------|-------|
| **Linus** (technical precision) | Code has architectural/performance issues | "You're holding the lock for the entire I/O. Did you test under load?" |
| **Ramsay** (standards-driven quality) | Tests or quality is subpar | "15 tests, 14 happy path. Where's the edge cases?" |
| **Bach** (BS detection) | Architecture decisions seem cargo-cult | "Why are you using microservices for a single-user tool?" |

**Default mode**: Linus for code review, Ramsay for test review.
**Calibration**: Level 1 (Direct) — professional and actionable.

#### Review Structure (per finding)

```markdown
## What's Broken
[Surgical description — specific file, line, function]

## Why It's Wrong
[Technical explanation with evidence, not opinion]

## What Correct Looks Like
[Clear model of excellence — code example or pattern reference]

## How to Fix It
[Actionable steps, specific to this codebase]

## Why This Matters
[Impact if not fixed — security, performance, maintainability]
```

#### Review Dimensions (weighted)

| Dimension | Weight | Mode | Questions |
|-----------|--------|------|-----------|
| **Correctness** | 30% | Linus | Does it do what requirements say? Edge cases handled? |
| **Security** | 20% | Linus | Injection risks? Auth bypasses? Data exposure? |
| **Maintainability** | 20% | Ramsay | Readable? Follows patterns? Future-proof? |
| **Performance** | 15% | Linus | N+1 queries? Unnecessary allocations? Missing indexes? |
| **Consistency** | 15% | Ramsay | Matches codebase conventions? Naming consistent? |

#### Review Process

1. **Read every changed file** completely
2. **Check against requirements** — is anything missing or extra?
3. **Check against ADR decisions** — does code match chosen approach?
4. **Check against architecture** — do components match diagrams?
5. **Check against QCSD ideation risks** — are `{QUALITY_RISKS}` from Step 3.5 mitigated?
6. **Flag issues** with severity:
   - 🔴 **BLOCKER** — must fix before merge (Linus mode: "This is completely broken")
   - 🟡 **WARNING** — should fix, but not blocking (Ramsay mode: "This is RAW")
   - 🔵 **SUGGESTION** — nice to have improvement

### 4. Multi-Agent Review Panel (L/XL)

For L/XL features, spawn 3 parallel review agents using brutal-honesty modes:

| Agent | Role | Mode | Model | Focus |
|-------|------|------|-------|-------|
| Agent 1 | Correctness Reviewer | Linus | sonnet | Requirements coverage, logic errors, performance |
| Agent 2 | Security Reviewer | Linus | sonnet | OWASP top 10, auth, data exposure |
| Agent 3 | Quality & Architecture Reviewer | Ramsay | sonnet | Test quality, pattern compliance, ADR adherence |

Each agent produces independent findings using the brutal-honesty structure
(What's Broken → Why → Correct → Fix → Impact). Synthesize into single report.

### 5. Gap Detection Loop (M+)

After review findings are compiled, run a gap check:

```
ITERATION {N}:
  1. Check: Every FR-{N} from requirements → has passing test or verification
  2. Check: Every ADR decision → reflected in code
  3. Check: Every critical risk from QCSD → has mitigation in code
  4. Check: Every 🔴 BLOCKER → has fix plan

  If gaps found:
    → List gaps explicitly
    → Fix gaps (return to Step 7 for code changes if needed)
    → Re-run smoke tests
    → Increment iteration counter

  If no gaps:
    → Proceed to acceptance criteria validation

  Max iterations: 3. After 3 iterations, flag remaining gaps for user decision.
```

This loop implements the "did we miss something?" recursive check from the original pipeline spec.

### 6. Acceptance Criteria Validation

For each requirement from Step 1:
```
FR-{N}: {requirement}
  Status: ✅ PASS | ❌ FAIL | ⚠️ PARTIAL
  Evidence: {how it was verified}
  Test: {test name or manual verification}
```

### 7. QE Report

Compile all findings into a structured report:

```markdown
# QE Report: {feature name}

## Summary
- Tests: {passed}/{total} ({percentage}%)
- Review findings: {blockers} 🔴 {warnings} 🟡 {suggestions} 🔵
- Requirements coverage: {covered}/{total} ({percentage}%)

## Test Results
### Unit Tests
{details}

### Integration Tests (L/XL)
{details}

## Review Findings
{list of findings with severity}

## Acceptance Criteria
{FR-N status table}

## Verdict
✅ READY FOR MERGE | ❌ NEEDS FIXES | ⚠️ CONDITIONAL APPROVAL
```

## Quality Gates

- [ ] All smoke tests pass
- [ ] No 🔴 BLOCKER findings remain
- [ ] All MUST requirements have ✅ PASS status
- [ ] Test coverage adequate for tier (S: smoke, M: unit, L: unit+integration, XL: full)
- [ ] Brutal-honesty review completed (M+) with structured findings
- [ ] Gap detection loop completed with zero remaining gaps (M+)
- [ ] QCSD quality risks from Step 3.5 verified as mitigated (M+)
- [ ] QE report created and complete
