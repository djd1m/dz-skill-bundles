# Step 3.5: QCSD Ideation Swarm

> Quality criteria assessment using HTSM v6.3, Risk Storming, and Testability analysis on refined ADR before implementation begins.

## Source

Adapted from [agentic-qe qcsd-ideation-swarm](https://github.com/proffesor-for-testing/agentic-qe) (Tier 3: Verified).

> Load: `references/agentic-qe/qcsd-ideation-swarm.md`

## When

M+ tiers. This step runs AFTER Step 3 (ADR) and BEFORE Step 6 (Implementation Plan).
- **M:** Core agents only (3 agents)
- **L/XL:** Core + conditional agents (3-9 agents based on feature flags)

## Model

sonnet (analytical quality assessment — multiple parallel agents)

## Input

- `{REQUIREMENTS}` from Step 1
- `{ADR_DECISIONS}` from Step 3 (with shift-left validation results)
- `{RESEARCH_FINDINGS}` from Step 2 (if L/XL)
- `{DOMAIN_MODEL}` from Step 4 (if available)

## Protocol

### 1. Flag Detection (MANDATORY)

Scan feature requirements + ADR decisions and SET these flags:

```
□ HAS_UI = FALSE
  Set TRUE if feature contains: UI, frontend, visual, component, screen, page, form,
  button, modal, dialog, dashboard, widget, interface, display, view, layout, CSS, styling

□ HAS_SECURITY = FALSE
  Set TRUE if feature contains: auth, security, credential, token, encrypt, PII,
  compliance, password, login, session, OAuth, JWT, permission, role, access control

□ HAS_UX = FALSE
  Set TRUE if feature contains: user experience, UX, journey, usability, user flow,
  persona, friction, delight, onboarding, retention, engagement
```

Output flag detection results before proceeding.

### 2. Spawn Core Agents (3 parallel — MANDATORY)

ALL THREE core agents MUST be spawned in a SINGLE message:

#### Agent 1: Quality Criteria Recommender (HTSM v6.3)

Analyze ALL 10 HTSM categories with weight and testability score:

| Category | Focus |
|----------|-------|
| Capability | Does it do what's required? |
| Reliability | Under stress, load, edge cases |
| Usability | Learnability, accessibility, efficiency |
| Charisma | Visual design, responsiveness |
| Security | Authentication, authorization, data protection |
| Scalability | Growth handling, resource limits |
| Compatibility | Browsers, devices, APIs, integrations |
| Performance | Speed, throughput, resource usage |
| Installability | Deployment, configuration, migration |
| Development | Testability, maintainability, portability |

**Output**: Quality criteria analysis with weighted scores per category.

#### Agent 2: Risk Assessor (SFDIPOT)

Apply SFDIPOT risk framework:
- **S**tructure: Code architecture, component dependencies
- **F**unction: Business logic, calculations, workflows
- **D**ata: Data integrity, migrations, edge values
- **I**nterfaces: API contracts, integration points
- **P**latform: OS, browser, infrastructure dependencies
- **O**perations: Deployment, monitoring, maintenance
- **T**ime: Scheduling, concurrency, timeouts, caching

**Output**: Minimum 10 risks with probability (1-5), impact (1-5), and risk score.

#### Agent 3: Requirements Validator (Testability Scoring)

Apply 10 Principles of Testability, score each 0-100:
1. Observability — can we see what the system does?
2. Controllability — can we put it in the state we need?
3. Decomposability — can we test parts independently?
4. Simplicity — is it only as complex as needed?
5. Stability — does it change predictably?
6. Information — do we have what we need to test?
7. Communication — do we know what to test?
8. Relation to prior work — can we leverage past tests?
9. Time — do we have enough time to test?
10. Accountability — who owns quality?

**Output**: Testability score (0-100) with per-principle breakdown.

### 3. Spawn Conditional Agents (L/XL, based on flags)

| Flag | Agent | Analysis |
|------|-------|----------|
| HAS_UI | Accessibility Auditor | WCAG 2.2 AA compliance assessment |
| HAS_SECURITY | Security Auditor | STRIDE threat modeling |
| HAS_UX | QX Partner | User journey and experience quality |

### 4. GO/NO-GO Decision

After all agents complete, apply decision logic:

| Metric | NO-GO | CONDITIONAL | GO |
|--------|-------|-------------|-----|
| Testability Score | < 40 | 40-79 | ≥ 80 |
| HTSM Coverage | < 6/10 categories | 6-7/10 | ≥ 8/10 |
| AC Completeness | < 50% | 50-89% | ≥ 90% |
| Critical Risks | > 2 | 1-2 | 0 |

- **NO-GO**: BLOCK. Requirements/ADR need rework. Return to Step 1 or Step 3.
- **CONDITIONAL**: Proceed with warnings. Pass findings to Steps 6-8 as advisory context.
- **GO**: Full confidence. Proceed to Step 6.

### 5. Synthesis Report

Generate consolidated `03.5_ideation_report.md` with:
1. Executive summary (GO/CONDITIONAL/NO-GO verdict)
2. Quality criteria scores (10 HTSM categories)
3. Risk matrix (top 10 risks by risk score)
4. Testability assessment (10 principles with scores)
5. Test ideas (categorized: unit / integration / E2E)
6. Conditional agent findings (if any)
7. Recommendations prioritized by risk × impact

## Output

Create `features/<slug>/03.5_ideation_report.md` with full QCSD report.

Set `{IDEATION_VERDICT}` variable (GO | CONDITIONAL | NO-GO).
Set `{QUALITY_RISKS}` variable with top risks for downstream steps.

## Quality Gates

- [ ] All 3 core agents spawned and completed
- [ ] Conditional agents spawned for all TRUE flags
- [ ] GO/NO-GO decision applied correctly
- [ ] NO-GO blocks pipeline (returns to Step 1 or 3)
- [ ] Testability score computed (0-100)
- [ ] Risk matrix contains ≥10 risks
- [ ] HTSM 10 categories all evaluated
- [ ] Ideation report created

## Anti-Patterns

| Anti-Pattern | Detection | Fix |
|-------------|-----------|-----|
| Skipping ideation "for speed" | No 03.5 artifact | BLOCK — mandatory for M+ |
| Fewer than 3 core agents | Agent count < 3 | BLOCK — all 3 required |
| Skipping conditional agents | TRUE flag without agent | BLOCK — spawn if flag is TRUE |
| Ignoring NO-GO verdict | Proceeding despite NO-GO | BLOCK — rework required |
| Placeholder analysis | "[TODO]" in report | Reject — full analysis required |
