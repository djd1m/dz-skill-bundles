# Coverage Formula — как считать % покрытия

> **Цель:** сделать оценку покрытия воспроизводимой и сопоставимой между запусками скилла. Два разных анализа одного и того же сценария должны давать близкий %.

---

## Базовая формула

```
coverage_raw = (N_full × 1.0 + N_partial × 0.5 + N_none × 0.0) / N_total
coverage_pct = round(coverage_raw × 100 / 5) × 5
```

Где:
- `N_full` — число **строк маппинга** с оценкой ✅ **Полное**
- `N_partial` — число **строк маппинга** с оценкой ⚠️ **Частичное**
- `N_none` — число **строк маппинга** с оценкой ❌ **Нет**
- `N_total = N_full + N_partial + N_none` — общее число строк маппинга

**Важно про под-шаги.** Если один логический шаг разделяется на несколько под-шагов (например, «Validate: 4a. ИНН через Контур.Фокус ✅ / 4b. ЭЦП ❌») — каждый под-шаг считается отдельной строкой маппинга. Это правильно, потому что под-шаги реально имеют разное покрытие и должны влиять на итоговый %. Правило «7–12 логических шагов» в SKILL.md касается фазы **декомпозиции** (Phase 2); после маппинга (Phase 3) итоговое `N_total` может быть больше — типично 10–18 строк.

**Округление до 5%** — принципиально. Точность оценки не больше этого в силу субъективности присвоения категорий. Никаких «73.4%» — только 70%, 75%, 80%.

---

## Определения категорий (однозначные)

### ✅ Полное (вес 1.0)

Шаг закрывается **готовым сервисом или комбинацией сервисов AI Factory** без:
- написания кастомных коннекторов
- развёртывания собственных моделей
- fine-tuning
- внешних non-AI-Factory API

**Примеры ✅:**
- «Отправить сообщение в Max» → MCP Мессенджер Max
- «Поиск в корпоративной БЗ» → Managed RAG
- «Классификация обращения» → FM (GigaChat) с промптом
- «Создать тикет в Jira» → MCP Jira
- «Выгрузить данные в Excel» → MCP Excel

### ⚠️ Частичное (вес 0.5)

Платформа закрывает **50–80% шага**, но требуется **одно из**:
- Разворачивание готовой open-source модели в ML Inference (без fine-tuning)
- Написание кастомного промпта для FM под специфичную задачу
- Small custom node (HTTP-call) в AI Workflows
- Light customization / сложная настройка

**Примеры ⚠️:**
- «Распознавание изображений в документах» → мультимодальная FM (Qwen-VL) справится с простыми, но не с плохими сканами
- «A/B тестирование» → AI Workflows умеет ветвление, но нет statistical framework
- «Push-уведомления» → нет MCP, но можно добавить через HTTP-узел
- «Генерация изображений» → нет MCP для image-gen, но можно деплоить SDXL через ML Inference
- «Генерация кода» → FM генерирует, но нет MCP к Git для ревью PR

### ❌ Нет (вес 0.0)

Один или несколько пунктов из списка:
- Нет MCP к нужной внешней системе, и она критична для шага
- Нужен fine-tuning на корпоративных данных (а не только prompting)
- Нужна модальность, для которой нет модели в FM и нужен деплой **со значительной кастомизацией** (не просто «развернуть HF»)
- Нужна детерминированная бизнес-логика, которую LLM-only не обеспечит с приемлемым risk-level
- Out-of-scope для AI/ML-стека (криптография, low-level system tooling)

**Примеры ❌:**
- «Интеграция с 1С:Бухгалтерия» — нет MCP, нужен кастомный шлюз
- «Загрузка в рекламный кабинет VK Ads» — нет MCP
- «Проверка ЭЦП» — out-of-scope
- «OCR сложных сканов с печатями» — нужен специализированный OCR, деплой + обёртка в MCP — это уже кастом
- «3-way match счёт/заказ/акт с финансовыми последствиями» — LLM-only рискован

---

## Правила однозначности (при сомнении)

1. **Колеблешься между ✅ и ⚠️?** → выбирай **⚠️**.
2. **Колеблешься между ⚠️ и ❌?** → выбирай **❌**.
3. **Honest assessment beats optimistic.** Пользователь заплатит за оптимизм в продакшне.

---

## Modifiers (опциональные коррекции)

После расчёта базовой формулы можно применить модификаторы, если есть особые обстоятельства:

### Modifier A: критичность gap'а

Если хотя бы один ❌-шаг является **критическим** (без него сценарий не работает вообще — например, OCR для обработки сканов), применяется:
```
coverage_pct_adjusted = min(coverage_pct, 65%)
```

Это отражает реальность: если критический шаг не покрыт, итоговое «high coverage» вводит в заблуждение.

**Пример:** сценарий «Обработка счетов». Даже если 9 из 10 шагов на ✅, но OCR сканов (шаг 2) — ❌ и это критично (без него не работают 90% счетов), то coverage ограничивается 65%, а не считается как 90%.

### Modifier B: зрелость интеграций

Если в сценарии **более 3 шагов** требуют кастомных MCP-разработок, применяется:
```
coverage_pct_adjusted = coverage_pct - 5%
```

Это отражает стоимость и риски интеграционной работы.

### Modifier C: compliance-критичность

Если сценарий подпадает под обязательный compliance (ФЗ-152 ПДн, маркировка рекламы ОРД, финансовое законодательство) и нет явного Validate-шага на ✅:
```
coverage_pct_adjusted = coverage_pct - 10%
```

### Правило применения модификаторов

Модификаторы применяются **последовательно**, но **итоговое значение также округляется до 5%**.

```
final = round(coverage_pct_after_modifiers / 5) × 5
```

## Примеры расчёта

### Пример 1: Агент поддержки

| Шаг | Оценка |
|---|---|
| 1. Ingest | ✅ |
| 2. Normalize PII | ✅ |
| 3. Classify | ✅ |
| 4. Retrieve | ✅ |
| 5. Generate | ✅ |
| 6. Decide routing | ✅ |
| 7. Create ticket | ✅ |
| 8. Notify | ✅ |
| 9. Log | ✅ |
| 10. Feedback loop | ✅ |

- N_full = 10, N_partial = 0, N_none = 0
- coverage_raw = (10 × 1.0) / 10 = 1.0 → 100%
- round(100/5) × 5 = 100%

Но если в реальности, например, ingest из IP-телефонии и нет STT — шаг 1 становится ⚠️, и:
- N_full = 9, N_partial = 1 → (9 + 0.5) / 10 = 0.95 → 95%

### Пример 2: Обработка счетов

| Шаг | Оценка |
|---|---|
| 1. Ingest | ✅ |
| 2. OCR сканов | ❌ (критический!) |
| 3. Extract fields | ✅ |
| 4a. Validate ИНН | ✅ |
| 4b. Validate ЭЦП | ❌ |
| 5. Match с договором | ⚠️ |
| 6. Detect exceptions | ✅ |
| 7. Prepare package | ✅ |
| 8. Route to 1С | ❌ |
| 9. Log | ✅ |

- N_full = 6, N_partial = 1, N_none = 3, N_total = 10
- coverage_raw = (6 + 0.5) / 10 = 0.65 → 65%
- round(65/5) × 5 = 65%
- Применяется **Modifier A** (критический OCR) → min(65%, 65%) = 65%
- Применяется **Modifier B** (3 шага требуют кастома — 4b, 5, 8) → 65% - 5% = 60%
- Итог: **60%**

### Пример 3: Маркетинговые кампании

| Шаг | Оценка |
|---|---|
| 1. Segment | ✅ |
| 2. Research | ✅ |
| 3. Plan | ✅ |
| 4a. Create text | ✅ |
| 4b. Create images | ⚠️ |
| 5. Personalize | ✅ |
| 6. A/B setup | ⚠️ |
| 7a. Launch email | ✅ |
| 7b. Launch push | ⚠️ |
| 7c. Launch рекламные кабинеты | ❌ |
| 8. Monitor | ✅ |
| 9. Optimize | ✅ |
| 10. Report | ✅ |

- N_full = 9, N_partial = 3, N_none = 1, N_total = 13
- coverage_raw = (9 + 1.5) / 13 = 0.808 → 80.8%
- round(80.8/5) × 5 = 80%
- **Modifier C** применяется если есть маркировка ОРД без явного ✅-шага валидации → 80% - 10% = 70%
- Итог: **70%** (или 65% при активных модификаторах)

## Интерпретация итоговых значений

| % покрытия | Интерпретация | Рекомендация |
|---|---|---|
| 90–100% | Почти идеально покрывается | Реализация = сборка из готовых блоков, минимум кастома |
| 75–90% | Хорошее покрытие | Гибридная архитектура: AI Factory + 1–2 кастомных коннектора |
| 60–75% | Среднее покрытие | Стоит трезво оценить ROI: 30–40% работы — кастомная разработка |
| 45–60% | Слабое покрытие | AI Factory — лишь часть решения. Нужен серьёзный кастом-стек |
| < 45% | Неподходящий кейс | Пересмотреть выбор платформы или разбить сценарий на подсценарии |

## Reproducibility checklist

Чтобы два запуска на одном и том же сценарии давали согласованный результат:

- [ ] Использован один и тот же каталог (фиксировать версию/дату каталога)
- [ ] Декомпозиция на одно и то же число шагов (используй decomposition-checklist — она даёт стабильные 10 пунктов)
- [ ] Использованы правила однозначности при сомнении
- [ ] Округлено до 5%
- [ ] Применены (или не применены) одни и те же модификаторы

В итоговом отчёте **всегда показывай таблицу с N_full / N_partial / N_none** и применёнными модификаторами — это делает оценку проверяемой.
