# Decomposition Checklist — 10 пунктов для любого сценария

> **Цель:** ни один шаг workflow не должен быть забыт. Пройди по всем 10 пунктам для каждого сценария. Если пункт не применяется — честно напиши «N/A», не пропускай молча.

---

## Чек-лист

### 1. Ingest — откуда приходит вход?

Варианты, которые часто забывают:
- Email (SMTP/IMAP)
- Мессенджеры (Max, Telegram, WhatsApp, VK)
- Веб-формы / чат на сайте
- API от клиентского приложения
- Watcher папки в S3 / FileSystem
- Запросы из тикет-системы (Jira/Trello/Service Desk)
- IP-телефония / SIP (если есть звонки)
- Upload пользователем (PDF/DOCX/XLSX/изображения)
- CDC из БД
- Очереди (Kafka/RabbitMQ)
- Scheduled (cron, time-based triggers)

**Вопросы для Claude:**
- Один канал или омниканально?
- Формат: текст, файл, изображение, аудио?
- Нужна ли буферизация/очередь?

### 2. Normalize — что нужно сделать перед основной логикой?

- Детекция языка
- Приведение к единому формату (plain text, markdown)
- PII-маскирование (ПДн, карты, СНИЛС, ИНН физлиц)
- NER (Named Entity Recognition)
- OCR если на входе скан/фото
- STT если на входе аудио
- Удаление разметки HTML/markdown
- Deduplication

**Вопросы:**
- Есть ли compliance-требования (ФЗ-152 ПДн)?
- Нужен ли audit-log до обработки?

### 3. Classify / Route — нужна ли категоризация?

- Тип обращения (вопрос/жалоба/запрос/baкансия)
- Тематика / категория
- Приоритет
- Тональность (sentiment)
- Intent detection
- Язык
- Маршрут обработки (кто ответит: бот/оператор/эксперт/менеджер)

**Вопросы:**
- Многоклассовая классификация или бинарная?
- Нужна ли confidence score и пороги?

### 4. Retrieve — нужен ли поиск корпоративных данных?

- Поиск в базе знаний
- Поиск по прошлым похожим кейсам (case-based reasoning)
- Поиск в документации, регламентах, политиках
- Поиск в договорах / актах
- Поиск в CRM / ERP
- Поиск в Confluence / вики

**Вопросы:**
- Насколько часто меняется корпус? (ежедневно / раз в месяц)
- Нужен ли гибридный поиск (semantic + keyword)?
- Какой размер корпуса?

### 5. Reason / Generate — какая основная LLM-логика?

- Ответ пользователю
- Резюмирование
- Извлечение структурированных данных (JSON extraction)
- Сравнение / matching
- Генерация черновика документа
- Reasoning с tool use
- Планирование
- Генерация кода / SQL / регулярки
- Генерация креативов (текст/изображение)

**Вопросы:**
- Нужно ли reasoning, или достаточно single-shot?
- Какой размер контекста нужен?
- Нужна ли цепочка агентов (multi-agent)?

### 6. Validate — какие проверки корректности?

- Проверка структуры ответа (schema)
- Бизнес-правила (суммы, даты, лимиты)
- Compliance (ПДн, маркировка рекламы, санкции)
- Сверка с мастер-данными (контрагенты, договоры, номенклатура)
- Проверка подписи (ЭЦП)
- Проверка дубликатов
- Safety checks (не пропустить токсичный контент в коммуникацию)

**Вопросы:**
- Что делать при fail? (retry/escalate/drop)
- Human-in-the-loop обязателен?

### 7. Act — что делаем во внешних системах?

- Создание тикета в Jira/Trello
- Обновление записи в CRM (amoCRM)
- Запись в БД (PostgreSQL, MongoDB)
- Сохранение файла в S3
- Вызов API внешней системы
- Отправка сообщения в чат/email
- Планирование follow-up
- Триггер в CI/CD
- Операция в Kubernetes / облаке

**Вопросы:**
- Идемпотентно ли действие?
- Нужны ли compensation-транзакции при ошибке?

### 8. Notify — как коммуницируем с людьми?

- Уведомление клиента (email / мессенджер / push / SMS)
- Уведомление оператора / эксперта
- Уведомление руководителя при эскалации
- Запрос подтверждения (approval flow)
- Статус-апдейты по ходу обработки

**Вопросы:**
- Какие SLA на уведомление?
- Есть ли quiet hours?

### 9. Log / Audit — что фиксируем?

- Все входящие запросы
- Все принятые решения (с обоснованием от LLM)
- Все действия во внешних системах
- Метрики качества (confidence, retries, escalations)
- Токены / стоимость
- Feedback от пользователей

**Вопросы:**
- Требуется ли иммутабельный лог (для compliance)?
- Нужен ли PII-redaction в логах?
- Где храним: PostgreSQL / S3 / специализированная система?

### 10. Feedback / Learn — как улучшается система?

- Сбор явного фидбэка (👍/👎)
- Сбор неявного (resolved/escalated, time-to-close)
- Периодическая ре-классификация ошибок
- A/B-тестирование промптов / моделей
- Fine-tuning на собранных данных
- Обновление базы знаний
- Дашборды качества

**Вопросы:**
- Есть ли цикл regular review?
- Кто отвечает за улучшения?

---

## Как использовать

1. Прочитай описание сценария от пользователя
2. Пройди по 10 пунктам и заполни применимые
3. Получишь список 7–12 шагов workflow
4. Переходи к Phase 3 (Mapping) из основного SKILL.md

## Пример применения (на сценарии «Агент поддержки»)

| # | Пункт | Применим? | Что конкретно |
|---|---|---|---|
| 1 | Ingest | ✅ | email + мессенджер + веб-чат |
| 2 | Normalize | ✅ | PII-маска, детекция языка |
| 3 | Classify | ✅ | тип (вопрос/жалоба), тематика, приоритет |
| 4 | Retrieve | ✅ | поиск в FAQ + регламенты |
| 5 | Reason/Generate | ✅ | черновик ответа с цитатами |
| 6 | Validate | ✅ | проверка что не раскрыты ПДн, тональность |
| 7 | Act | ✅ | создание тикета в Jira |
| 8 | Notify | ✅ | ответ клиенту, уведомление оператора |
| 9 | Log | ✅ | запись всех обращений + решений в PostgreSQL |
| 10 | Feedback | ✅ | 👍/👎 от клиента + time-to-resolve |

Итого: 10 шагов workflow для маппинга.
